    Mass Transcode — Purga de "Pudrición" en Lote

    "Si la AMD no muere, muere por CPU. Nada escapa."

    Mass Transcode es una herramienta oculta (no expuesta en TUI) para re-encodificar archivos MKV problemáticos en lote, aprovechando aceleración de hardware con fallback automático a CPU libx264.

    Esta es la contrapartida en cadena de montaje de UniversalRescuer. Mientras que UniversalRescuer procesa archivos individuales con la "Escalera de Resiliencia" de 4 niveles, Mass Transcode es un "mata-rata químico" simple: intenta AMD VA-API, si muere, pasa a CPU sin preguntas.

    🎯 Caso de Uso

    Tienes una carpeta con 500 MKVs de diferentes trackers, muchos con corrupción menor: - Cabeceras WinX - Audio desincronizado - Subtítulos en formatos raros - Stream metadata corrupto

    Quieres procesarlos en paralelo sin intervención manual, aceptando calidad "suficiente" en lugar de "perfecta".

    Mass Transcode es tu aliado.

    🚀 Uso

    Preparación: Lista de Rutas

    1. Crea un archivo de texto con las rutas a procesar: 

      cat > /app/logs/rutas_host.txt << 'EOF'
/run/media/user/storage/Películas/1.mkv
/run/media/user/storage/Películas/2.mkv
/run/media/user/storage/Series/season1/ep01.mkv
EOF

    2. Ejecuta desde dentro del contenedor: 

      python3 mass_transcode.py

    Ubicación por defecto del archivo de entrada: /app/logs/rutas_host.txt

    ⚙️ Lógica de Procesamiento

    Flujo Por Archivo

    Entrada: archivo.mkv
    ↓
[INTENTO 1: AMD VA-API]
    ffmpeg -vaapi_device /dev/dri/renderD128 \
           -i archivo.mkv \
           <hw_args_amd> \
           archivo_provisional.mkv
    ↓
¿Éxito? ✅ Reemplaza original y continúa
        ❌ Fallo → Borra provisional
    ↓
[INTENTO 2: CPU libx264 (Fallback)]
    ffmpeg -i archivo.mkv \
           -c:v libx264 -preset fast -crf 22 \
           -c:a aac -b:a 128k \
           -fflags +genpts+igndts+discardcorrupt \
           archivo_provisional.mkv
    ↓
¿Éxito? ✅ Reemplaza original y continúa
        ❌ Fallo máximo → Borra provisional, siguiente archivo

    Argumentos Específicos

    AMD VA-API: 

    [
    'ffmpeg', '-hide_banner', '-loglevel', 'error', '-y',
    '-vaapi_device', '/dev/dri/renderD128',
    '-i', ruta,
    *hw_args_list,  # Generado por HardwareAgent
    '-c:a', 'aac', '-b:a', '128k',
    '-map', '0:v:0', '-map', '0:a?', '-map', '0:s?',
    '-ignore_unknown', '-map_metadata', '-1',
    '-max_interleave_delta', '0',
    output
]

    Notas: - -max_interleave_delta 0: Evita errores de interleave con pistas desincronizadas. - -ignore_unknown: Ignora streams extraños (data, attachment, etc.). - -map_metadata -1: Limpia metadatos globales de spam.

    CPU libx264 (Fallback): 

    [
    'ffmpeg', '-hide_banner', '-loglevel', 'error', '-y',
    '-i', ruta,
    '-c:v', 'libx264', '-preset', 'fast', '-crf', '22',
    '-c:a', 'aac', '-b:a', '128k',
    '-map', '0:v:0', '-map', '0:a?', '-map_metadata', '-1',
    '-fflags', '+genpts+igndts+discardcorrupt',
    output
]

    Comparación AMD vs CPU: - AMD: Más rápido (~1-3h para 50GB, si la GPU responde) - CPU: Más lento (~4-8h), pero 100% compatible

    📊 Output y Reporte

    No hay TUI fancy. El output es raw:

    🚀 RECON: Motor AMD listo para la batalla.
🎬 [1/525] Iniciando: Fucker_MKV_2k_HEVC_EP1.mkv
   ⚡ Intentando aceleración AMD...
   ✅ Éxito con AMD.
🎬 [2/525] Iniciando: Corrupto_WinX.mkv
   ⚠️ La AMD ha gripado con la pudrición. Saltando a CPU...
   🚜 Rescatado por CPU con éxito.
🎬 [3/525] Iniciando: Irrecuperable.mkv
   ⚠️ La AMD ha gripado con la pudrición. Saltando a CPU...
   💀 FALLO TOTAL: Ni la CPU ha podido con este cadáver.

    Archivos que fallan completamente: - Se omiten (no se reemplazan) - Se logean en stdio - El proceso continúa

    🔗 Interacción con UniversalRescuer

    • Mass Transcode: Procesamiento en lote, rápido, 2 intentos.
    • UniversalRescuer: Procesamiento individual, 4 niveles sofisticados, más lento.

    Cuándo usar cuál:

    Escenario Herramienta
    500 archivos, algo de "pudrición", tolerancia moderada Mass Transcode
    1-10 archivos críticos, máxima calidad requerida UniversalRescuer (God Mode)
    Identificación de archivos rotos antes de procesarlos UniversalRescuer (Verifier)

    📝 Limitaciones Conocidas

    1. Sin validación post-procesamiento: A diferencia de UniversalRescuer, Mass Transcode no re-valida el archivo tras re-encod. Confía en que si FFmpeg no reporta error, el archivo es OK.

    2. Sin atomicidad sofisticada: Si el proceso crash a mitad, los archivos parcialmente procesados pueden quedar como archivo_provisional.mkv. Revisa /app/RawLoadrr/tmp/ manualmente.

    3. Output mínimo: Solo ve el progreso en consola. Para logging persistente, redirige stdout: 

      python3 mass_transcode.py > /app/logs/mass_transcode_$(date +%s).log 2>&1

    🛡️ Seguridad de Datos

    Punto crítico: os.replace(output, ruta) es atómico a nivel del filesystem en Linux. - Si el comando falla, el original queda intacto. - El provisional se borra. - Cero riesgo de pérdida accidental.

    Sin embargo: - Si tu línea de comando tiene un typo y especifica rutas equivocadas, los archivos equivocados pueden reemplazarse. Revisa rutas_host.txt dos veces.

    📡 Performance y Paralelismo

    Actual: Procesamiento secuencial (1 archivo por vez).

    PENDING_RECON: ¿Implementar multiprocessing para 4 procesos ffmpeg paralelos? Requería: - Queue thread-safe - Gestión de locks para /dev/dri - Testing con VA-API compartida entre procesos

    Actualmente no implementado porque VA-API en Docker puede ser inestable si múltiples ffmpeg intentan usar la GPU a la vez.
