    🔑 Conceptos Técnicos Documentados

    1. La Escalera de Resiliencia (Niveles 3-0)

    Nivel 3: VapourSynth + Hardware
    - Ubicación: /app/RawLoadrr/tmp/TEMP_RESCUE/ - Plugin: L-SMASH-Works (compila) - Comando exacto: vspipe -c y4m "script.vpy" - | ffmpeg ... <hw_args>

    Nivel 2: Remux Médico
    - Estrategia: Audio/subs re-encode, video copy - Comando exacto: ffmpeg -fflags +genpts -i input -map 0 -c:v copy -c:a aac -af aresample=async=1

    Nivel 1: Re-Encode Bruto + Fallback CPU
    - Intento HW: ffmpeg ... <hw_args> - Trigger fallback: "Could not write header" en stderr - CPU Tank: ffmpeg ... -c:v libx264 -fflags +genpts+igndts+discardcorrupt

    Nivel 0: Salvaguarda
    - Acción: 🛑 REQUIRES_MANUAL_REVIEW: <ruta>

    2. HardwareAgent (Detección Automática)

    Plataformas: - NVIDIA → h264_nvenc / hevc_nvenc - AMD → h264_vaapi / hevc_vaapi (con fallback a CPU si vainfo falla) - Intel → h264_qsv / hevc_qsv - CPU → libx264 / libx265

    Ubicación del código: /app/MKVerything/modules/hardware_agent.py

    3. Arquitectura de Volúmenes (Docker Compose v1.6)

    FAST_WORK_DIR: /app/RawLoadrr/tmp/TEMP_RESCUE - Archivos temporales (NVMe, rápido) - Provantídeo scripts .vpy, temporales de re-encod

    Mapeo de volúmenes: 

    - ./work_data/tmp:/app/RawLoadrr/tmp:z          # ← FAST_WORK_DIR
- ./work_data/logs:/app/logs:z                  # ← rescue_process.log
- /run/media/${USER}:/run/media/${USER}:z       # ← Almacenamiento masivo

    4. Patología de "Pudrición" Multimedia

    6 tipos documentados: 1. Corrupción de Cabecera: "Could not write header" → Nivel 3/2 2. Desincronización de Audio: → Nivel 2 (aresample=async=1) 3. Subtítulos Dañados: → Nivel 2 (conversión a SRT) 4. Aspect Ratio Roto (WinX): → Nivel 1 CPU (-metadata:s:v:0 sar=1/1) 5. Stream Extra Corrupto: → -ignore_unknown, -map_metadata -1 6. Interleave Error: → -fflags +genpts, -max_interleave_delta 0

    5. Atomicidad y Validación

    Patrón Write-Validate-Replace: 1. Procesar en TEMP_RESCUE (temp) 2. Validar con Verifier (4 capas: mkvmerge, ffprobe, ffmpeg null, metadata) 3. os.replace() atómico → original intacto si falla

    Garantías formales: - Sin pérdida de datos (original nunca tocado hasta validación) - Sin corrupción silenciosa (4-capa validation exhaustiva) - Sin archivos parciales (atomicidad OS-level)

    Comandos FFmpeg (Exactamente como en código)

    Nivel 3: 

    vspipe -c y4m "/app/RawLoadrr/tmp/TEMP_RESCUE/script.vpy" - | \
ffmpeg -hide_banner -y -i - -i "<input>" -map 0:v -map 1:a? -map 1:s? <hw_args> "<output>"

    Nivel 2: 

    ffmpeg -hide_banner -y -fflags +genpts -i "<input>" \
  -map 0 -c:v copy -c:a aac -af aresample=async=1 -c:s srt "<output>"

    Nivel 1 (HW): 

    ffmpeg -hide_banner -y -i "<input>" -map 0 <hw_args> "<output>"
# Si trigger="Could not write header" en stderr → fallback a:

    Nivel 1 (CPU Tank): 

    ffmpeg -hide_banner -y -i "<input>" \
  -c:v libx264 -preset fast -crf 22 \
  -c:a aac -b:a 128k -c:s srt \
  -map 0:v:0 -map 0:a? -map 0:s? \
  -ignore_unknown -map_metadata -1 \
  -metadata:s:v:0 sar=1/1 \
  -fflags +genpts+igndts+discardcorrupt "<output>"

    Mass Transcode (AMD): 

    ffmpeg -hide_banner -loglevel error -y \
  -vaapi_device /dev/dri/renderD128 \
  -i "<input>" <hw_args_amd> \
  -c:a aac -b:a 128k \
  -map 0:v:0 -map 0:a? -map 0:s? \
  -ignore_unknown -map_metadata -1 \
  -max_interleave_delta 0 "<output>"

    📝 Notas de Desarrollador (PENDING_RECON)

    Áreas de Incertidumbre Marcadas

    1. MOZ_X11_EGL: Variable en Dockerfile, pero ¿se necesita realmente en tu setup? Marcar como PENDING_RECON.

    2. Cross-Filesystem Atomicidad: Si TEMP_RESCUE y el destino están en filesystems diferentes, os.replace() puede no ser seguro. Necesita shutil.move() + fsync().

    3. Mass Transcode Paralelismo: Sin multiprocessing actualmente. ¿Implementar 4 procesos paralelos? Requiere testing con VA-API compartida.

    4. Reporte de "Fallidos": Mass Transcode no guarda lista de archivos que fallaron completamente. Manual review necesaria.

    5. Bitstream Scan Lentitud: Validación Capa 3 puede tomar 1+ hora para archivos de 100GB. ¿Saltarla si capas 1-2 pasan?
