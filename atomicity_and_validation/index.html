    Atomicidad y Validación — Garantías de Integridad

    "No es suficiente rescatar un archivo. Debe vivir correctamente."

    Este documento explica cómo Singularity Core garantiza que los datos procesados no se corrompen mediante operaciones atómicas y validación exhaustiva.

    🔐 Garantía Fundamental: Reemplazo Atómico

    El Patrón: Write-Validate-Replace

    Todos los procesos de re-encod siguen este flujo:

    1. Procesar en FAST_WORK_DIR   ← /app/RawLoadrr/tmp/TEMP_RESCUE
     (archivo_provisional.mkv)

2. Validar resultado           ← Verifier (4 capas)

3. Reemplazo atómico          ← os.replace() a nivel OS

    Por Qué No Sobrescribir Directamente?

    ❌ Peligro: Si ffmpeg falla a mitad del re-encod, sobrescribió el original con basura. 

    # MAL:
output_path = original_path  # ← Suicidio de archivo
subprocess.run(f"ffmpeg ... -i original_path {output_path}")
# Si ffmpeg crash: original perdido

    ✅ Seguro: 

    temp_path = "/app/RawLoadrr/tmp/TEMP_RESCUE/archivo_temp.mkv"
output_path = original_path

# Procesa en temp
subprocess.run(f"ffmpeg ... {temp_path}")

# Valida
if verifier.validate(temp_path):
    os.replace(temp_path, output_path)  # ← Atómico
else:
    os.remove(temp_path)  # ← Original intacto

    Atomicidad a Nivel del Filesystem

    os.replace(src, dst) en Linux es atómico:

    os.replace(temp_path, output_path)

    Garantía: - Una de dos cosas sucede: O el reemplazo completo ejecuta, O nada sucede. - Jamás queda en estado mitad-reemplazado. - Jamás ambos archivos existen simultáneamente (excepto por microsegundos).

    Limitación: - Debe estar en el mismo filesystem. Si temp_path está en NVMe y output_path en HDD externo, la operación puede no ser tan atómica.

    Solución en docker-compose: 

    volumes:
  - ./work_data/tmp:/app/RawLoadrr/tmp:z  # Mismo "volumen host"
  - /run/media/...:/run/media/...:z       # Mapeo consistente

    📋 La Validación en 4 Capas (Verifier)

    Antes de cualquier reemplazo, el archivo debe pasar 4 pruebas independientes:

    Capa 1: Integridad de Contenedor (mkvmerge)

    # verifier.py
result = subprocess.run(["mkvmerge", "-i", file_path], capture_output=True)
if result.returncode != 0:
    return False  # Contenedor MKV está roto

    Qué valida: - Estructura interna del contenedor EBML es válida - Todos los segmentos están presentes - CRC de cluster (si está habilitado) es correcto

    Tiempo: ~1-5 segundos

    Capa 2: Integridad de Streams (ffprobe)

    result = subprocess.run(
    ["ffprobe", "-v", "error", "-show_streams", file_path],
    capture_output=True
)
streams = json.loads(result.stdout)

    Qué valida: - Cada stream (video, audio, subs) es legible - Codec existe y es soportado - Dimensiones, frame rate, bitrate son razonables

    Tiempo: ~2-10 segundos

    Capa 3: Decodificación Bitstream (ffmpeg null sink)

    result = subprocess.run(
    f"ffmpeg -hide_banner -i {file_path} -f null -",
    shell=True,
    capture_output=True
)
if result.returncode != 0:
    return False  # Hay errores de bitstream

    Qué valida: - Llama a los decodificadores reales (libx264, libvpx, etc.) - Busca frames corruptos, bits inválidos, comprensión incorrecta - Si hay errores de bitstream, ffmpeg los reporta y falla

    Tiempo: ~30% del tiempo total del archivo (~15 segundos para 1h de contenido)

    Nota crítica: Esta es la prueba más lenta pero más confiable. Un archivo puede pasar capas 1-2 pero fallar aquí si el video en sí es ilegible.

    Capa 4: Coherencia Metadata (Size/Duration Ratio)

    original_info = get_mediainfo(original_file)
processed_info = get_mediainfo(processed_file)

size_ratio = processed_info.filesize / original_info.filesize
duration_ratio = processed_info.duration / original_info.duration

# Si el tamaño cambió drasticamente, algo malo pasó
if size_ratio < 0.3 or size_ratio > 3.0:
    return False

    Qué valida: - El archivo procesado tiene duración similar (±1%) - El nuevo tamaño es razonable (re-encod espera ~1-2x el original) - Detección de "truncamiento silencioso" (ffmpeg escribió solo 10 segundos y cerró sin error)

    Tiempo: ~negligible

    🔄 Flujo Completo: Entrada a Salida Segura

    graph TD
    Start[Archivo Original] --> L3["🪜 Nivel 3\n(VapourSynth)"]

    L3 -->|Éxito| Temp3["archivo_temp.mkv\nen FAST_WORK_DIR"]
    L3 -->|Fallo| L2["↪️ Nivel 2\n(Remux)"]

    Temp3 --> Val3["✅ Validar\n(4 Capas)"]
    Val3 -->|Fallo| Cleanup3["❌ Borrar provisional\ny reintentar"]
    Cleanup3 --> L2

    Val3 -->|Éxito| Replace3["🔄 os.replace\n(Atómico)"]
    Replace3 --> End["✅ Archivo Seguro"]

    L2 -->|Éxito| Temp2["archivo_temp.mkv"]
    L2 -->|Fallo| L1["↪️ Nivel 1\n(Re-Encode)"]

    Temp2 --> Val2["✅ Validar\n(4 Capas)"]
    Val2 -->|Éxito| Replace2["🔄 os.replace"]
    Val2 -->|Fallo| Cleanup2["❌ Borrar"]
    Cleanup2 --> L1

    L1 -->|Éxito| Temp1["archivo_temp.mkv"]
    L1 -->|Fallo| L0["↪️ Nivel 0\n(Salvaguarda)"]

    Temp1 --> Val1["✅ Validar\n(4 Capas)"]
    Val1 -->|Éxito| Replace1["🔄 os.replace"]
    Val1 -->|Fallo| Cleanup1["❌ Borrar"]
    Cleanup1 --> L0

    L0 --> Manual["🛑 REQUIRES_MANUAL_REVIEW"]

    Puntos críticos: 1. Cada nivel temporizador tiene su propia validación antes de tocar el original. 2. Si validación falla, el temp se borra y se reintem ta el siguiente nivel. 3. Si todos fallan, el original queda intacto y se marca para revisión.

    🛡️ Escenarios de Fallo y Recuperación

    Escenario A: Crash Durante Re-Encod

    ffmpeg está procesando...
→ Power failure
→ archivo_temp.mkv quedó truncado
→ Al reiniciar MKVerything...

    Qué sucede: 1. UniversalRescuer reinicia con el mismo archivo 2. Detecta que archivo_temp.mkv existe en TEMP_RESCUE → lo borra 3. Reintenta desde Nivel 3

    Original: ✅ Intacto (nunca fue tocado)

    Escenario B: Validación Falla Post-Encod

    ffmpeg terminó sin error
archivo_temp.mkv se creó
Validación Capa 3 (bitstream scan) detecta corrupción

    Qué sucede: 

    if not verifier.validate(temp_path):
    os.remove(temp_path)      # Borra temp
    # Reintentar siguim nivel

    Original: ✅ Intacto (nunca fue tocado)

    Escenario C: Filesystem Casi Lleno

    TEMP_RESCUE tiene 512MB libres
ffmpeg intenta escribir 50GB
→ Queda espacio, pero comprimido

    Qué sucede: 1. FFmpeg falla por "Disco lleno" → returncode != 0 2. UniversalRescuer detecta el error 3. Borra el temp parcial 4. Reintentar (probablemente siguiente nivel)

    Original: ✅ Intacto

    Escenario D: os.replace() Falla (Filesystem Cross-Boundary)

    temp_path = /app/RawLoadrr/tmp/... (volumen dockerizado)
output_path = /run/media/external/... (HDD USB)
os.replace(temp, output) → EXDEV error

    Actualment e: PENDING_RECON. El código no maneja esto explícitamente.

    Solución futura: 

    try:
    os.replace(temp_path, output_path)
except OSError as e:
    if e.errno == errno.EXDEV:
        # Cross-filesystem: usar shutil.move + sync
        shutil.move(temp_path, output_path)
        os.fsync(...)
    else:
        raise

    📊 Validación Post-Procesamiento

    Capa 4 Detallado: Comparación Metadata

    Tras cada rescate exitoso, el log registra:

    [2026-03-22 16:15:44]    Capa 4 (Comparación):
[2026-03-22 16:15:44]      Original: 1h 22min 50sec, 1.2GB
[2026-03-22 16:15:44]      Procesado: 1h 22min 49sec, 1.1GB
[2026-03-22 16:15:44]      Duration delta: 0.1% (✅ OK)
[2026-03-22 16:15:44]      Size ratio: 0.92 (✅ OK, dentro 0.3x-3.0x)
[2026-03-22 16:15:44]    ✅ ÉXITO en Nivel 1

    Umbrales: - duration_delta < 1%: Archivo procesado tiene la misma longitud - size_ratio 0.3-3.0: Cambio de tamaño razonable para recodificación - Si violas estos, marcar para revisión manual

    🧬 Garantías Formales

    Garantía Cómo Prueba
    Sin pérdida de inicio de archivo Procesa en temp, reemplaza atómicamente Escenario A
    Sin datos parciales 4-capa validation antes de reemplazo Escenario B
    Sin corrupción silenciosa Bitstream scan + coherencia metadata Verifier capas 3-4
    Sin duplicados corruptos Original intacto si todo falla Escenario C, D

    ⚠️ Limitaciones y PENDING_RECON

    1. Validación Capa 3 es lenta: Para archivos de 100GB, el bitstream scan puede tomar 1+ hora. ¿Saltarla si el archivo pasa capas 1-2?

    2. Cross-Filesystem Atomicidad: Si temp y destino están en filesystems diferentes, os.replace() no es garantizado atómico.

    3. Corrupción de Hardware: Si la RAM/CPU tiene errores transitorios (bit flip), nada nos lo dirá. Mitigación: checksums SHA256 pre/post.

    4. Timezone/NTP Issues: Los timestamps puede incompletos si el reloj del contenedor es incorrecto. PENDING_RECON: sincronizar NTP en init.

    🔍 Debugging Manual

    Si desconfías de la validación:

    # Dentro del contenedor:
python3 << 'EOF'
from modules.verifier import Verifier
v = Verifier()
result, details = v.full_validation("/app/RawLoadrr/tmp/mi_archivo.mkv", verbose=True)
print(f"Resultado Final: {result}")
print(f"Detalles: {details}")
EOF

    Esto te mostrará exactamente qué capa falla.
