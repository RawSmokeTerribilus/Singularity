    Troubleshooting — Patología de "Pudrición" Multimedia

    "Cuando un archivo es más cadáver que video, aquí es donde vienen a morir en paz."

    Esta guía detalla los errores típicos de corrupción ("pudrición") que encuentran los usuarios y cómo la Escalera de Resiliencia los mitiga automáticamente.

    🔍 Anatomía de la Pudrición Multimedia

    Tipos Principales

    1. Corrupción de Cabecera (Header Corruption)

    Síntomas: 

    ffmpeg error: "Could not write header"
ffmpeg error: "Invalid data found"
ffmpeg error: "Unknown format"

    Causa: El contador de frames, timestamps o metadatos de contenedor están dañados. Ocurre típicamente cuando: - El archivo se descargó incompleto - El antivirus escaneó y corrupted el archivo - Edición de WinX Media o Windows Media Player (escribe timestamps inválidos)

    Cómo lo mitiga el sistema:

    Nivel Estrategia Resultado
    Nivel 3 L-SMASH bypassa el header del contenedor, lee el stream directamente a bajo nivel ✅ Rescatado (si el stream en sí está OK)
    Nivel 2 ffmpeg -fflags +genpts regenera los timestamps completos ⚠️ Audio posiblemente desincronizado
    Nivel 1 Re-encod completo: escribe un header nuevo desde cero ✅ Rescatado pero más lento

    Código relevante: 

    # universal_rescuer.py - Nivel 1 detección
fallback_trigger="Could not write header"
if result == "FALLBACK_TRIGGERED":
    # CPU tank con correcciones:
    # -metadata:s:v:0 sar=1/1: corrige aspect ratio roto por WinX
    # -fflags +genpts: regenera PTS

    2. Desincronización de Audio (Audio Out of Sync)

    Síntomas: 

    Los labios no coinciden con el audio
ffmpeg reports "skipped samples" durante decodificación

    Causa: El stream de video y audio tienen diferentes timestamps o frame rates: - El audio se editó pero el video no - DTS vs PTS mismatch - Contenedor con sellos de tiempo inválidos

    Cómo lo mitiga:

    Nivel Estrategia
    Nivel 3 VapourSynth con LSMASH mantiene sincronización videonativa; luego ffmpeg re-sincroniza
    Nivel 2 aresample=async=1 resampling automático; fflags +genpts regenera PTS de sync point
    Nivel 1 (CPU) Copia el stream original, solo audio AAC re-encode a 128k (manteniendo PTS original)

    Código exacto (Nivel 2): 

    ffmpeg -fflags +genpts -i input.mkv \
  -c:v copy -c:a aac \
  -af aresample=async=1 \  # ← Resampling sincrónico
  -c:s srt output.mkv

    3. Subtítulos Dañados o en Formato Incompatible

    Síntomas: 

    mkvmerge: UTF-8 validation error
ffmpeg: Invalid UTF-8 in subtitle
Subtítulos en SSA/ASS con errores de codificación

    Causa: - Subtítulos con encoding corrupto (CP1252 vs UTF-8 confundidos) - Formato ASS con etiquetas rotas - Subtítulos incrustados con bugs de mkvtoolnix antiguo

    Cómo lo mitiga:

    Nivel Estrategia
    Nivel 3 Ignora subs, re-encoda solo video+audio
    Nivel 2 Convierte a SRT puro (decodifica ASS a texto plano ASCII-safe)
    Nivel 1 -c:s srt + map 0:s? intenta incluir subs, fallback a ignorarlas

    Nota: Nivel 2 es el especialista en subs rotos. Si llega a Nivel 1, probablemente los subs se pierden.

    4. Aspect Ratio Roto (WinX/MediaPlayer Corruption)

    Síntomas: 

    Video estirado (4:3 mostrado como 16:9)
Video aplastado (16:9 mostrado como 4:3)
ffprobe muestra values SAR="0/1" o inválidos

    Causa: Windows Media Player o WinX escriben metadatos inválidos de aspect ratio.

    Cómo lo mitiga:

    Nivel 1 (CPU Tank específicamente): 

    -metadata:s:v:0 sar=1/1  # ← Resetea SAR a neutral (Fix WinX)

    Si llegas a Nivel 1 CPU, se aplica este fix automáticamente.

    PENDING_RECON: ¿Debería aplicarse también en Nivel 2 o Nivel 3?

    5. Stream Extra Corrupto ("Attachment", "Data", etc.)

    Síntomas: 

    ffmpeg: Unknown stream type @0xa (type 10)
ffmpeg: Stream index 10 not found
mkvmerge: Invalid codec ID

    Causa: El contenedor MKV tiene streams "attachment" (fuentes, imágenes, scripts) con codec inválido o corrupto.

    Cómo lo mitiga:

    Nivel Estrategia
    Nivel 3 -map 0:v -map 1:a? -map 1:s? extrae solo V+A+S, ignora otros
    Nivel 2 -map 0 mapea TODO pero ffmpeg puede fallar en stream 10
    Nivel 1 -map 0:v:0 -map 0:a? -map_metadata -1 limpia metadatos y attachments

    Código: 

    # universal_rescuer.py - Nivel 1 fallback
"-map_metadata", "-1",
"-ignore_unknown",  # ← Ignora streams desconocidos

    6. Interleave Error (Pistas Desordenadas)

    Síntomas: 

    ffmpeg: Too many packets buffered for output stream
ffmpeg: DTS back values < PTS

    Causa: El contenedor tiene frames/paquetes fuera de orden temporal. Ocurre en: - Edición manual with mkvmerge sin re-muxing - Archivos grabados en tiempo real con seekback - Corrupción parcial de índice

    Cómo lo mitiga:

    Nivel 2: 

    -fflags +genpts  # ← Regenera PTS de cero

    Nivel 1: 

    -max_interleave_delta 0  # ← Desactiva el buffer de interleave en Mass Transcode
-fflags +genpts+igndts+discardcorrupt

    🔧 Solución Manual: Cuando Nivel 0 No Rescata

    Si todos los niveles fallan, tienes opciones manuales:

    Opción A: ForceKeyframes + Corruption Discard

    ffmpeg -hide_banner -y \
  -fflags +bitexact+discardcorrupt \
  -i corrupted.mkv \
  -c:v libx265 -preset ultrafast \
  -c:a aac -b:a 64k \
  -max_muxing_queue_depth 1024 \
  rescue4.mkv

    Rationale: Desactiva validaciones estrictas y ignora frames corruptos. Resultado: Vídeo probablemente con artefactos o saltos, pero "jugable".

    Opción B: Extracción y Re-Mux

    # Extrae streams individuales
ffmpeg -i corrupted.mkv -c:v copy stream_video.h264
ffmpeg -i corrupted.mkv -c:a copy stream_audio.aac

# Re-muxea en contenedor limpio
ffmpeg -i stream_video.h264 -i stream_audio.aac \
  -c copy -map 0:v -map 1:a output.mkv

    Rationale: Si el contenedor está roto pero los streams son OK, esto puede rescatarlos.

    Opción C: Aceptación

    Si nada funciona, el archivo probablemente está irrecuperable. Considera: - Descargarlo de nuevo desde el tracker original - Solicitar un reencod en el foro - Aceptar un downgrade de calidad (ej: source HDTV en lugar de remux)

    📝 Logs y Diagnóstico

    Dónde Buscar Errores

    UniversalRescuer: 

    /app/logs/rescue_process.log
    Línea por línea, con exactitud de qué comando falló y por qué.

    Mass Transcode: 

    Stdout/Stderr en tiempo real (sin persistencia automática)
    Redirige manualmente si necesitas logging: 
    python3 mass_transcode.py >> /app/logs/mass_transcode_$(date +%s).log 2>&1

    devices:

    ✅ Checklist: Antes de Reportar "Archivo Irrecuperable"

    1. ¿El archivo pasó Verifier correctamente? Si sí, probablemente NO necesita rescate.
    2. ¿Corriste God Mode? (MKVerything Launcher → Opción 5). Si no, el archivo no fue auditor procesado.
    3. ¿Verificaste los logs? /app/logs/rescue_process.log dirá exactamente qué falló.
    4. ¿El Nivel 0 reporta REQUIRES_MANUAL_REVIEW? Si no, quizás el rescate sí funcionó pero no te diste cuenta.
    5. ¿Reproduciste el archivo rescatado en VLC? 90% de "fallos" son falsos positivos; VLC "entiende" más que ffmpeg.
